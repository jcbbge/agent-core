# Mapping Adjacent Possibilities

## When to use this skill

Use this skill when you need to:
- **Prioritize multiple possible projects** - Which work unlocks the most future opportunities?
- **Evaluate strategic decisions** - What becomes possible if I solve this problem?
- **Plan learning investments** - Which skills compound and transfer?
- **Assess opportunity value** - Is this a platform or point solution?
- **Strategic planning** - What sequence of work creates compounding advantages?
- **Feel overwhelmed by options** - Which choice preserves the most optionality?

**Trigger phrases:** "Should I work on X or Y?", "What should I build next?", "Is this worth learning?", "How should I prioritize?"

## Core principles

Adjacent-possibilities thinking reveals what becomes accessible when constraints are removed or problems are solved. It maps the **opportunity space** that opens up from current actions.

**Five core lenses:**

1. **Dominoes** - What does solving X enable you to solve next?
2. **Constraint removal** - What becomes possible if Y is no longer limiting?
3. **Compounding effects** - What builds on what?
4. **Unexpected unlocks** - What non-obvious things become easier?
5. **Opportunity mapping** - What's now in reach that wasn't before?

**Key distinction:** This prevents **local optimization** (solving isolated problems) and enables **strategic positioning** (work that unlocks multiple future opportunities).

## Quick start: The basic workflow

When evaluating a decision, apply this sequence:

**Step 1: Identify the immediate outcome**
- What does this work directly produce or enable?
- What constraint does it remove?
- What capability does it build?

**Step 2: Map second-order effects**
- What does that immediate outcome make possible?
- What new problems can you now solve?
- What new opportunities can you now pursue?
- What skills/assets/resources do you gain?

**Step 3: Trace to third order**
- What do those second-order possibilities enable?
- Look for compounding patterns
- Identify platform vs point solutions
- Note learning transfer potential

**Step 4: Evaluate strategic value**
- Compare option value vs immediate value
- Assess reversibility (can you undo this choice?)
- Identify preserved optionality
- Consider resource constraints

**Reality check:** Map 2-3 levels deep, not infinite chains. Focus on probable adjacencies, not fantasy dominoes.

## Core frameworks

### 1. Platform vs Point Solution

**Platform opportunities:**
- Create foundations for multiple future projects
- Generate compounding advantages
- Enable network effects
- Produce transferable capabilities

**Point solutions:**
- Solve one specific problem
- Provide immediate value
- Limited future leverage
- Isolated outcomes

**Example:**
- **Point:** "Build a custom WordPress plugin for one client"
- **Platform:** "Build a SaaS tool for WordPress agencies" (serves many clients, compounds with each user)

### 2. Option Value Assessment

Evaluate work by what it **keeps open** vs what it produces:

**High option value:**
- Reversible decisions
- Preserves multiple future paths
- Builds general capabilities
- Low commitment required

**Low option value:**
- Irreversible commitments
- Locks you into specific path
- Narrow specialization
- High switching costs

**Barbell strategy:** Combine safe, stable work (protects downside) with high-optionality experiments (preserves upside).

### 3. Learning Transfer Analysis

Assess skills by how broadly they apply:

**High transfer:**
- Core concepts that apply across domains
- Fundamental capabilities (systems thinking, debugging)
- Platform technologies (databases, APIs)
- Problem-solving patterns

**Low transfer:**
- Tool-specific knowledge that doesn't generalize
- Domain-locked expertise
- Obsolescence-prone skills
- Narrow techniques

**Example:**
- **High transfer:** Learning TypeScript (applies to any TS codebase, transfers to typed languages)
- **Low transfer:** Learning specific WordPress theme framework (only applies to WordPress, framework-specific)

### 4. The Dominoes Pattern

Map sequential unlocking:

```
Action → Immediate unlock → Second-order unlock → Third-order unlock
```

**Example: VPS Migration Decision**
- Action: Migrate to VPS
- Immediate: Control infrastructure, learn Linux admin
- Second-order: Can offer DevOps consulting, build products requiring custom infrastructure
- Third-order: Higher-margin consulting work, ability to build self-hosted SaaS products

**Key question:** Which first domino tips over the most valuable next dominoes?

### 5. Constraint Removal Framework

Identify binding constraints and model removal:

**Step 1:** What's currently preventing you from pursuing X?  
**Step 2:** What becomes possible if that constraint is removed?  
**Step 3:** What does removing the constraint require?  
**Step 4:** Is removing this constraint a platform move?

**Example:**
- Constraint: "Can't build real-time features" (lacking WebSocket knowledge)
- Removal enables: Streaming overlays, live collaboration tools, real-time dashboards
- Platform effect: Real-time capability unlocks entire product category

## Practical application patterns

### Pattern 1: Project prioritization

When choosing between multiple projects:

**For each option, ask:**
1. What immediate problem does this solve?
2. What capabilities does this build?
3. What does it enable me to do next?
4. Does this compound with existing work?
5. How reversible is this choice?

**Strategic heuristic:** Prefer work that:
- Unlocks multiple future opportunities
- Builds transferable capabilities
- Creates platform effects
- Preserves optionality

### Pattern 2: Learning investment evaluation

When deciding what to learn:

**Assess:**
- **Transfer breadth:** How many domains can I apply this to?
- **Compounding potential:** Does this make learning other things easier?
- **Unlock threshold:** Does this remove a binding constraint?
- **Option value:** Does this preserve multiple career paths?

**Example: TouchDesigner learning evaluation**
- Transfer: Real-time systems, visual programming, creative coding (applies to streaming, installations, live events)
- Compounding: Each project builds reusable node networks
- Unlocks: Entire category of creative tech work
- Option value: Can pivot to creative coding, live events, interactive installations

**Red flags:**
- Tool-specific knowledge with narrow application
- Skills with short half-life (rapid obsolescence)
- Deep specialization with high switching costs
- Learning that doesn't transfer or compound

### Pattern 3: Strategic sequencing

When planning a sequence of work:

**Map the dependency tree:**
- What needs to come first?
- What builds on what?
- Where are the compounding points?
- What's the optimal sequence?

**Sequencing principles:**
- Do platform work before point solutions
- Build capabilities before products that use them
- Learn transferable skills before specialized applications
- Preserve optionality early, commit later

**Example: Developer roadmap**
```
Phase 1: Infrastructure (platform)
→ Learn cloud architecture, Docker, CI/CD
→ Unlocks: Can build/deploy any application type

Phase 2: Core product (uses platform)
→ Build SaaS with learned infrastructure
→ Unlocks: Revenue, customer base, domain expertise

Phase 3: Expansion (compounds)
→ Offer infrastructure consulting
→ Build tools for similar developers
→ Leverage: Both technical skills and domain knowledge compound
```

## Working with real constraints

Adjacent-possibilities thinking must account for actual limitations:

**Time constraints:**
- Can't pursue all adjacent possibilities simultaneously
- Must sequence and prioritize
- Some adjacencies require pivots (can't be parallel paths)

**Resource constraints:**
- Cash flow may force point solutions over platforms
- Learning time competes with earning time
- Must balance exploration with exploitation

**Energy constraints:**
- Context switching has cognitive cost
- Pursuing too many adjacencies spreads focus thin
- Must maintain execution momentum

**Validation requirement:**
- Not all adjacent possibilities are valuable
- Some "adjacent" opportunities are actually distant
- Must validate demand/feasibility

**Strategic balance:** Map adjacencies to make informed choices, then focus execution on chosen path. Don't let possibility mapping prevent commitment.

## Decision-making examples

### Example 1: VPS vs Overlays vs Contract Work

**User context:** Developer with limited time, needs income, multiple possible projects

**VPS Migration:**
- Immediate: Control infrastructure, cost savings
- Second-order: DevOps consulting capability, self-hosted products possible
- Third-order: Higher-margin work, independence from managed platforms
- Platform effect: Infrastructure capability applies to all future projects
- Option value: Reversible (can move back), preserves multiple paths

**Streaming Overlays:**
- Immediate: Real-time system experience, WebSocket mastery
- Second-order: Can build live-data products, interactive applications
- Third-order: Creative tech opportunities, live event production
- Transfer: Real-time patterns apply broadly
- Option value: Builds capability without major commitment

**Contract Work:**
- Immediate: Cash flow, client relationship
- Second-order: Reputation, potential repeat work
- Third-order: Possible long-term client, steady income
- Platform effect: Minimal (solves immediate problem only)
- Option value: Low (time committed, skills may not transfer)

**Strategic analysis:**
- Contract work provides immediate income but limited adjacencies
- VPS teaches infrastructure, which unlocks consulting and products (platform)
- Overlays teach real-time systems, which transfer broadly (platform)
- Optimal: Pursue contract work for cash flow while building VPS/overlays skills in parallel (barbell strategy)

### Example 2: Learning TouchDesigner

**User context:** Developer exploring creative coding for projection mapping

**Immediate capability:**
- Visual programming for real-time graphics
- Node-based workflow mastery
- Projection mapping technical skills

**Second-order adjacencies:**
- Real-time visual systems → streaming overlays
- Creative workflows → understanding creator tools
- Public creative coding → audience building
- Live event experience → event production opportunities

**Third-order possibilities:**
- Streaming overlay business (combines real-time + creative)
- SaaS for creators (understanding creative workflows)
- Creative tech consulting
- Live event production services

**Transfer analysis:**
- Real-time system concepts transfer to web applications
- Visual programming patterns apply to other node-based tools
- Creative technical skills differentiate from typical dev work

**Platform vs point:**
- **Platform:** Builds capability that applies to multiple product categories
- Unlocks: Creative tech, live events, real-time systems, creator tools
- Not point solution: Not just "can do projection mapping"

**Strategic value:** High transfer, multiple adjacencies, preserves optionality for creative tech career paths

### Example 3: Paper Craft Asset Workflow

**User context:** Building automated pipeline for 3D to 2D paper craft conversion

**Immediate:**
- Workflow automation for own projects
- 3D-to-2D translation skills
- Consistent visual style pipeline

**Second-order:**
- Can offer paper craft assets to game studios (service)
- Could package workflow as tool for others (product)
- Geometric abstraction skills transfer to motion graphics

**Third-order:**
- If service: Steady client work in niche market
- If product: Passive income from tool sales
- If skill: Apply to any geometric abstraction work

**Platform evaluation:**
- Service path: Point solution (trades time for money per client)
- Product path: Platform (tool scales without linear time investment)
- Skill path: High transfer (geometric abstraction applies broadly)

**Strategic decision:** The workflow automation itself is platform (reusable capability), but monetization paths have different platform effects. Product path has highest compounding potential.

## Common pitfalls

**Fantasy dominoes:** Mapping unrealistic chains of events (be probabilistically honest)

**Analysis paralysis:** Mapping possibilities without executing (must commit eventually)

**Ignoring constraints:** Treating all adjacencies as equally accessible (resource reality matters)

**Over-optimization:** Choosing work solely for adjacencies, ignoring immediate needs (cash flow matters)

**Infinite exploration:** Never committing because "better adjacencies might exist" (choose and commit)
